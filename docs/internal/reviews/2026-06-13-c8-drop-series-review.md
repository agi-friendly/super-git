## Findings

### P1 - undo reports success even when execution-record consumption fails

Location:

- `crates/super-git-core/src/git/undo_history_edit.rs:97`
- `crates/super-git-core/src/git/undo_history_edit.rs:102`
- `crates/super-git-core/src/git/undo_history_edit.rs:129`

After a successful history-edit undo, the code removes the execution record with
best-effort `fs::remove_file` and discards the error. The returned effects then
unconditionally say the record was consumed and the same plan can be executed
again.

That is not just a cosmetic mismatch. If the branch ref and working tree were
restored but record deletion fails, the user receives `{ ok: true }` while the
replay guard still exists. A fresh preview of the same state can produce the
same state-based `plan_id`, hit the same execution-record path, and be rejected
as `execution_already_attempted`. A stale undo token can also retain provenance
while the record remains on disk.

One reviewer reproduced this by making `.git/super-git/executions` non-writable
after execute: undo returned success, the record still existed, and re-execute
was rejected with `execution_already_attempted`.

Suggested fix: make record consumption part of the success contract. Either
delete/rename the record atomically before returning success, or return a
structured partial failure such as `undo_partial_failure` /
`undo_record_consume_failed` that says the ref and worktree are restored but the
record remains. Add a regression test where record deletion fails after the ref
and worktree restore.

### P1 - drop sync partial failures lose machine-readable recovery details

Location:

- `crates/super-git-cli/src/output.rs:78`
- `crates/super-git-core/src/error.rs:97`
- `crates/super-git-core/src/error.rs:122`
- `crates/super-git-core/src/git/execute_history_edit.rs:1112`
- `crates/super-git-core/src/git/undo_history_edit.rs:175`
- `docs/safety-model.md:161`

The new `ExecuteSyncPartialFailure` and `UndoSyncPartialFailure` variants carry
the recovery-critical fields: `branch_ref`, `observed_tip`, `sync_completed`,
`execution_record_path`, and `safe_next`. However, the CLI JSON envelope only
builds `error.details` for the older worktree `ExecutePartialFailure` variant.

For drop execute/undo sync failures, JSON-first consumers therefore receive the
stable `error.code`, but must parse prose in `error.causes` to recover the
observed tip, sync progress, and next safe action. That contradicts the safety
model's partial-failure contract and weakens the exact failure mode where agents
most need structured recovery data.

Suggested fix: extend `structured_error_details` to match both sync partial
failure variants and emit stable JSON for at least:

- `status: "failed_partial"`
- `action`
- `branch_ref`
- `observed_tip`
- `sync_completed`
- `execution_record_path`
- `cleanup.automatic_undo_available: false`
- `cleanup.safe_next`

Add CLI output tests that construct both execute and undo sync partial failures
and pin the JSON shape.

### P2 - `super-git.plan.v0.4` hash projection changed without a schema bump or compatibility path

Location:

- `crates/super-git-core/src/model.rs:32`
- `crates/super-git-core/src/model.rs:704`
- `crates/super-git-core/src/model.rs:845`
- `crates/super-git-core/src/git/preview_history_edit.rs:705`
- `crates/super-git-core/src/git/preview_history_edit.rs:708`
- `crates/super-git-core/src/git/preview_history_edit.rs:726`

The plan schema is still `super-git.plan.v0.4`, but the plan-id hash projection
now includes `prediction`, and `result_summary` now includes
`commits_dropped` with a serde default of `0`.

That means a tree-preserving `super-git.plan.v0.4` generated by `main` can
deserialize under the new binary, but recomputing the plan id can produce a
different digest because the new projection serializes fields that did not
exist in the old projection. The likely user-visible result is a
`plan_id_mismatch`, not an explicit schema/version incompatibility.

If invalidating old preview plans is the intended safety posture, the schema
should move to `super-git.plan.v0.5` or the execute error should clearly say the
plan belongs to an older hash contract and must be re-previewed. If v0.4
compatibility is intended, keep a v0.4 hash projection for old tree-preserving
plans and add a fixture generated from `main` to prove it.

### P2 - active docs still describe the old tree-preserving history-edit contract

Location:

- `docs/command-reference.md:481`
- `docs/architecture.md:115`
- `docs/architecture.md:266`
- `docs/roadmap.md:20`
- `docs/roadmap.md:240`
- `docs/safety-model.md:259`
- `docs/safety-model.md:275`

Several active docs still describe history-edit undo as branch-pointer-only or
working-tree-untouched. That is correct for tree-preserving edits, but no longer
true for `drop`, where undo requires a clean working tree, checks ignored-path
collisions against the pre-execute tip, restores the ref, and then synchronizes
the index and working tree.

There are also smaller command-surface mismatches: `docs/architecture.md` does
not list `preview history-edit`, `predict merge`, or `predict rebase` in the
current commands, and the roadmap still says the Stage 6 `drop` consumer is
open even though the C8-drop lifecycle has landed.

Suggested fix: split the docs language into tree-preserving history edits versus
tree-changing `drop`. Make the command list and roadmap status match the current
CLI surface.

### P3 - `preview history-edit` read-only wording hides synthetic object writes

Location:

- `docs/command-reference.md:258`
- `docs/command-reference.md:284`
- `crates/super-git-core/src/git/preview_history_edit.rs:115`
- `crates/super-git-core/src/git/conflict_prediction.rs:239`

`preview history-edit` is documented as read-only, but a drop preview calls
`predict_replay_onto`, and clean replay steps synthesize unreferenced commits
with `git commit-tree`. This does not touch refs, the index, config, or the
working tree, and the objects are gc-collectable. Still, it writes to the object
database.

The `predict rebase` docs already call out this nuance. `preview history-edit`
should either avoid object writes or use the same precise boundary:
refs/index/working-tree/config remain read-only, but conflict prediction may add
unreferenced, gc-collectable objects.
