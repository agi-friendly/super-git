# 임시 워크트리를 활용한 안전한 브랜치 최신화 아이디어

작성자: Codex 팀장

상태: 그냥 아이디어 메모. 아직 설계 아님.

## 금차장이 낸 아이디어

금차장이 평소에 워크트리를 많이 쓰면서 떠올렸던 아이디어.

현재 저장소에서 서버를 켜두고 작업 중일 때, 다른 브랜치를 최신 base로
리베이스하려면 보통 그 브랜치로 체크아웃해야 한다. 그러면 현재 저장소의
파일들이 잠깐 과거 상태로 바뀌고, 서버나 작업 환경이 꼬일 수 있다.

그래서 브랜치 리베이스를 현재 작업 저장소에서 직접 하지 말고,
워크트리에서 처리하면 훨씬 안전하지 않을까? 라는 생각.

예를 들어 리베이스해야 할 브랜치가 10개 있고:

- 5개는 이미 연결된 워크트리가 있음
- 5개는 브랜치만 있고 워크트리는 없음

이때:

- 워크트리가 있는 브랜치는 해당 워크트리에서 리베이스
- 워크트리가 없는 브랜치는 super-git이 임시 워크트리를 만들어서 거기서 리베이스
- 전부 끝나면 임시 워크트리는 제거

이런 식이면 현재 작업 중인 저장소는 건드리지 않고 여러 브랜치를 최신화할 수 있다.

## 내가 검토한 느낌

이건 가능성이 있다. 그리고 super-git의 방향과도 잘 맞는다.

다만 처음부터 "배치 리베이스"로 만들면 범위가 커진다. 리베이스는 충돌,
중단, continue, abort, skip, branch ref 복구 같은 복잡한 상태가 바로 따라온다.

그래서 첫 이름은 `safe branch refresh` 정도가 좋아 보인다.

처음에는 진짜 리베이스 전체를 다루기보다, 아래처럼 아주 안전한 케이스만
다루면 좋겠다:

- 대상 브랜치가 최신 base의 ancestor 위에 있음
- 대상 브랜치에 고유 커밋이 없음
- 대상 브랜치가 체크아웃된 워크트리가 있다면 그 워크트리가 clean
- 진행 중인 merge/rebase/cherry-pick 같은 operation이 없음

이런 조건이면 충돌 리베이스라기보다 "브랜치 ref를 최신 base로 안전하게
끌어올리는 작업"에 가깝다.

## 조심해야 할 점

이미 워크트리에 체크아웃된 브랜치를 최신화하면, 그 워크트리의 실제 파일은
바뀐다. 만약 그 워크트리에서 서버가 돌고 있다면 영향을 받을 수 있다.

그래서 preview에서는 반드시 이런 정보를 보여줘야 한다:

- 이 브랜치가 기존 워크트리에서 처리되는지
- 임시 워크트리에서 처리되는지
- 실제 파일 변화가 생기는 워크트리가 어디인지
- undo가 가능한지

워크트리가 없는 브랜치는 super-git이 임시 워크트리를 만들어 처리할 수 있으니
상대적으로 더 안전해 보인다.

## 나중에 상상하는 흐름

언젠가 만든다면 이런 식일 수 있다:

```bash
super-git preview branch-refresh --onto main --branches "works/*"
super-git execute --plan plan.json
super-git undo --token result.json
```

초기 MVP는 충돌 처리 없는 안전 케이스만.

그 다음 단계에서:

- 로컬 커밋이 있는 브랜치 rebase preview
- 임시 워크트리에서 실제 rebase 시도
- 충돌 시 임시 워크트리 보존
- conflict guide 연결
- 성공한 rebase의 제한적 undo

이런 식으로 확장하면 될 것 같다.

## 한 줄 결론

금차장이 말한 아이디어는 충분히 가능성이 있고, super-git이 이미 만든
`inspect -> preview -> execute -> undo` 기반과도 잘 맞는다.

다만 지금 당장 설계하지는 말고, 나중에 rebase 계열 기능을 다룰 때
`safe branch refresh`라는 작은 첫 범위로 다시 꺼내보면 좋겠다.
